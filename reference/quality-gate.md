# 质检门禁

每次输出前必须完成以下检查。

## 数据库检查

- 已确认关系类型存在、启用、非黑名单。
- 已确认实体规则与关系类型参考文档一致。
- 已读取现有 `published` 和 `pending_review` 实例。
- 已读取候选关系两端实体类型对应的 `nba_ko_entity_tags`。
- 已检查候选两端实体是否存在，匹配方式包括实体名、别名和外部 ID。
- 新候选没有与现有实例重复。

## 证据检查

- 每条候选至少有一个可访问来源 URL。
- 官方来源优先于二手来源。
- 证据片段必须直接支持三元组，不能只靠常识推断。
- 来源时间与关系时效匹配。

## 口径检查

- 不抽取关系类型参考文档明确排除的内容。
- 不把属性误建成关系。
- 不把别名误建成实体关系。
- 不把缺失实体伪装成已存在实体。
- 不为凑数量创建低价值候选。

## 输出检查

- 输出是合法 JSON。
- `relation_code` 与任务输入一致。
- 每条候选包含实体类型、实体名称、关系类型、证据、置信度。
- 每条候选包含 `entity_resolution`。
- 缺失实体必须标记 `needs_create = true`，并提供 `create_suggestion`。
- 若无新增，说明已检查范围和无新增原因。

## 人工审核提示

以下情况必须进入 `conflicts` 或 `update_candidates`，不得直接作为新增：

- 多个来源互相矛盾。
- 只有非权威来源支持。
- 实体名称可能是别名、旧名或翻译差异。
- 候选三元组任一端实体不存在，需要审核时新建。
- 关系需要时间范围但无法确认起止时间。
